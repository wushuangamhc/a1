# 01 工程初始化与骨架

## 目标

把当前仓库整理成可以持续迭代的 Dota 2 自定义游戏工程骨架，不改变共享协议命名。

## 涉及目录

- `package.json`
- `tsconfig.*.json`
- `webpack.panorama.config.js`
- `game/`
- `shared/`
- `content/panorama/`

## 验收标准

- TypeScript to Lua 输出目录明确
- Panorama 打包入口明确
- shared types 可被双端直接引用
- README 对地图实体命名约定描述完整

## 给 Kimi 的执行提示词

你现在是这个 Dota 2 Mod 项目的实现工程师。请基于当前仓库做工程化整理，但必须遵守以下限制：

1. 不要改动 shared 中已经存在的事件名、net table 名和核心类型名。
2. 优先保证 game、panorama、shared 三层依赖关系清晰。
3. 如果要补充脚本或配置，保持最小可行，不要一次引入过多外部工具。
4. 输出结果时说明你改了哪些文件、为什么改，以及还有哪些 Dota 本地环境依赖无法在当前仓库直接验证。

你的任务：
- 检查 build 脚本和 tsconfig 是否自洽
- 如果发现 panorama 或 tstl 的配置缺口，最小化补齐
- 不要删除现有 HUD/scoreboard/endgame 入口
- 完成后给出一份简短自测清单
